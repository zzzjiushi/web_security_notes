# Authentication(身份验证)

### 什么是authentication？

身份验证是确认用户或客户端身份的过程。网站可能暴露于任何连接互联网的用户面前，因此强大的身份验证机制对有效保障网络安全至关重要。

### 身份验证的三种主要类型：

- 知识因素：你所知晓的信息，例如密码或安全问题的答案
- 持有因素：你持有的物品，例如手机或安全令牌等实体物件
- 固有因素：你所具有的特征或行为，例如你的生物特征或行为模式

身份验证机制依赖于一些列技术来验证一个或多个因素

### 认证和授权的区别：

**身份验证**是确认用户身份真实性的过程。

**授权**则涉及验证用户是否被允许执行特定操作。

例如：身份验证用于确定使用用户名 `Carlos123` 的人是否确实是创建该账户的本人。而用户 `Carlos123` 通过身份验证后，其权限决定了可执行的操作范围。

### 身份验证漏洞的产生方式：

- 认证机制薄弱，因为它们无法有效防范**暴力破解**攻击。
- 实现种的**逻辑缺陷**或**编码不佳**会使认证机制被攻击者完全绕过。被称为“认证失效”

在许多网页开发领域，逻辑缺陷会导致网站表现异常，这可能使安全问题，也可能不是。然而，由于认证对安全至关重要，错误的认证逻辑很可能会让网站面临安全隐患。



## 认证机制中的漏洞

### 基于密码的login中的漏洞

对于采用基于密码登录流程的网站，用户可自行注册账户，或由管理员分配账户。该账户关联一个唯一的用户名和一个密码，用户需在登录表单中输入这些信息以完成身份认证。

要想通过此身份认证，有多种方式：

#### 暴力攻击

暴力破解攻击是指**攻击者通过反复尝试的方式猜测有效的用户凭证**。此类攻击通常采用自动化方式，利用包含用户名和密码的字典文件进行。自动化该过程——尤其是借助专用工具——可能使攻击者能够以极高速度进行海量登录尝试。

暴力破解并非总是单纯的对用户名和密码进行完全随机的猜测，攻击者还需结合基本逻辑或公开信息，对暴力攻击进行精细化调整。

**自动化反复尝试用户名+密码组合**

工具常见：

- burp intruder
- hydra
- ffuf
- 脚本

#### 暴力破解用户名

 用户名若遵循可识别的模式（如电子邮件地址），则特别容易被猜中。例如，常见的企业登录账号采用格式：`firstname.lastname@somecompany.com` 。然而，即使没有明显模式，有时候高权限账户也会使可预测的用户名创建，例如 `admin` 或 `adminstrator`。

**在审计过程中，需要检查网站是否公开纰漏潜在的用户名。**

- 企业邮箱模式
- 默认账户：admin,adminstrator,root,test....
- 文件泄露
- 注册接口返回差异



#### 暴力破解密码

密码同样可能遭受暴力破解，其难度取决于密码强度。许多网站采用某种密码策略，强制用户创建高熵密码——至少在理论上，这类密码仅凭暴力破解更难攻破。

用户往往不会创建由随机字符组合而成的强密码，而是选择自己能记住的密码，并试图将其硬塞进密码策略要求中。例如，**`mypassword`如果不允许使用 ，用户可能会尝试使用`Mypassword1!` 或`Myp4$w0rd` 等替代方案。**

对潜在凭证和可预测模式的了解意味着，暴力破解攻击往往比简单地遍历所有可能的字符组合更为复杂，因而更具成效。

#### 用户名枚举

用户名枚举是**指攻击者通过观察网站行为的变化，来判断某个用户名是否有效。**

**不同输入——不同响应——信息泄露**

用户名枚举通常发生在登录界面（例如输入有效用户名但密码错误时）或注册表单（当输入已被占用的用户名时）。这极大的缩短了暴力破解登录所需要的时间，因为攻击者能够快速生成有效用户名的候选列表。

在尝试暴力破解登录页面时，应特别注意以下方面的差异：

- **状态码：**在暴力破解攻击过程中，由于绝大多数猜测都是错误的，因此返回的HTTP状态码很可能对所有猜测都保持一致，但是 **若某次猜测返回了不同的状态码，则强烈表明该用户名是正确的。**

- **错误信息：**有时候返回的错误信息会因用户名和密码勇士错误，或仅密码错误而有所不同。

- **响应时间：**若多数请求的响应时间相近，任何偏离此规律的情况都暗示后台发生了异常。

  用户不存在——直接返回

  用户存在——需要比对hash，时间会更长



#### 有缺陷的暴力破解防护

暴力破解攻击极有可能需要经过多次失败尝试，攻击者才能成功攻破账户。

从逻辑上讲，暴力破解的核心在于 **尽可能增加自动化的难度，并降低攻击者尝试登录的频率。**

**防范暴力破解攻击的常见方式：**

- 若远程用户登录失败次数过多，则锁定其尝试访问的账户
- 若远程用户在短时间内进行过多尝试，则封禁其IP地址

##### 一，当IP被封禁时

在某些实现方案中，当IP所有者成功登录后，失败尝试次数的计数器就会重置。这就意味着攻击者只需要每尝试几次就登录自己的账户，就能永远避免触发该限制。

**实现方式：**在单词表中定期插入自己的登录凭证，就足以使这种防御措施形同虚设。

##### 二，账户锁定时——密码喷洒

> 突破口：
>
> 账户锁定只锁定账户，不是整个系统，所以从对密码的爆破，变成了对用户名的爆破，也就是准备少量的密码字典，准备大量的用户名字典，让密码去匹配用户名。

锁定账户能为特定账户提供一定程度的保护，防止针对性暴力破解攻击。然而，这种方法无法有效阻止攻击者试图获取任意随机账户的暴力破解攻击。

可以采用以下方法来规避此类保护机制：

1. 建立一份可能有效的候选用户名列表。这可以通过用户名枚举实现，或直接基于常见的用户名列表
2. 确定一个极小的密码候选列表，其中包含至少一位用户可能使用的密码。关键在于，所选密码数量不得超过允许的登录尝试次数。
3. 使用intruder工具，将选定的每个密码与候选用户名逐一组合尝试。这样便能对所有的账户进行暴力破解，同时避免触发账户锁定机制。只需要某个用户使用了三组密码中的任一一组，即可攻破该账户。

##### 三，用户速率限制

网站防范暴力破解攻击的另一种方式是实施用户速率限制。当用户在短时间内发起多登录请求时，其IP地址将被封禁。（基于用户速率的IP封禁）

通常情况下，IP地址只能通过以下方式之一解除封禁：

- 经过一定时间后自动
- 由管理员手动执行
- 用户在成功完成验证码后手动操作

该机制，可通过多种方式伪造其真实IP地址以绕过封锁，由于该限制基于用户IP地址发送的HTTP请求速率，因此有时候若能设法通过单次请求猜测多个密码，也可绕过此项防御机制。

> ##### 四，关于IP封禁
>
> 1. 基于失败次数计数模型
>
>    ```
>    if login_failed:
>        fail_count[ip] += 1
>    
>    if login_success:
>        fail_count[ip] = 0
>    
>    if fail_count[ip] > threshold:
>        block(ip)
>    ```
>
> 2. 用户速率限制，在短时间内发起多请求——封IP
>
>    ```
>    if requests_in_last_60_seconds[ip] > 10:
>        block(ip, 30 minutes)
>    ```
>
> 3. IP封禁的实现层级：
>
>    - 应用层封禁：
>
>      ```
>      blocked_ips[ip] = timestamp + 30min
>      if ip in blocked_ips:

>      ```
>
>      容易被绕过：伪造IP头
>
>    - WEB服务层（Nginx/Apache)

>      limit_req zone=login burst=3;

>
>      这属于，服务器级限速，不在业务代码中，更难绕过

#### HTTP基本认证

虽然已经很老旧，但HTTP基本认证因其相对简单且易于实现的特性，至今仍偶尔可见。

在HTTP基本认证中，客户端从服务器获取认证令牌——该令牌由用户名与密码拼接后经Base64编码生成。此令牌由浏览器存储管理，并在后续每次请求时自动添加至 `Authorization` 请求头中，格式如下：

```
Authorization:Basic base64(username:password)
```

这种认证方式不被视为安全，首先，它要求每次请求都反复发送用户的登录凭证。除非网站同时实施HSTS（强制安全传输策略），否则用户凭证极易在中间人攻击中被截获。

此外，HTTP基本认证的实现通常不支持暴力破解防护。由于令牌仅由静态值构成，这可能使其容易遭受暴力破解攻击。

HTTP基本认证尤其容易受到与会话相关的攻击（特别是跨站请求伪造攻击），其本身对此类攻击毫无防护能力。

在某些情况下，利用存在漏洞的HTTP基本认证可能仅使攻击者获得对看似无关紧要页面的访问权限。然而，此类认证信息不仅会扩大攻击面，还可能在其他更敏感的场景中被重复利用。

### 多因素认证中的漏洞

对于大多数网站，都是依赖单因素认证来验证用户身份，但是，也有部分网站要求用户通过多种认证因素来证明其身份。

基于“你所知”和“你所特”的强制性与可选性**双因素认证（2FA）**整日益普及。这种认证通常**要求用户同时输入传统密码和来自其特有的非网络物理设备生成的临时验证码。**

**比较重要的是**，多因素认证的优势是需要验证多种不同的因素才能实现。如果通过两种不同的方式验证同一种因素并不构成真正的双因素认证。

#### 双因素认证令牌

验证码通常由用户从某种实体设备上读取。如今许多高安全性网站会为用户提供专用设备，例如访问网上银行或工作笔记本电脑时使用的RSA令牌或键盘设备。

另一方面，部分网站会通过短信将验证码发送至用户的手机。虽然从技术层面这仍属于“你拥有的东西”的验证因素，但这种方式存在被滥用的风险。验证码不是由设备自身生成，而是由短信传输，可以使得验证码被截获，此外还存在SIM卡盗换风险。



#### 绕过双重验证

如果用户先被要求输入密码，然后在另一个页面输入验证码，那么在输入验证码之前，用户实际上已处于“登录状态”。这种情况下，可以测试是否在完成第一步身份验证后直接跳转到：仅限”登录用户“的页面。有时，某些网站在加载页面时并未实际验证用户是否完成第二步验证。



#### 存在缺陷的双因素验证逻辑

双因素认证中存在逻辑缺陷的情况时有发生：用户完成初始登录步骤后，网站**未能充分验证执行第二步操作的用户身份是否与初始登录这一致。**

**2FA身份绑定错误的核心问题：**第二步验证码验证时，没有确认当前验证的人和第一步验证的人是不是同一个用户。

例如：用户第一步使用常规凭据登录，具体如下：

```
POST /login-steps/first HTTP/1.1
host: xxx.com
...
username=carlos&password=qwerty
```

随后，系统会为其分配一个关联其账户的Cookie,然后进入流程的第二步：

```
HTTP/1.1 200 OK
Set-Cookie: account=carlos

GET /login-steps/second HTTP/1.1
Cookie: account=carlos
```

这个系统没有用session记录用户身份，而是把用户身份放在cookie中，完全信任cookie。

在这种情况下，攻击者可以使用自己的凭据登录，但在提交验证码时将`account`cookie的值修改为任意用户名。

```
POST /login-steps/second HTTP/1.1
Host: vulnerable-website.com
Cookie: account=victim-user
...
verification-code=123456
```

若攻击者能够暴力破解验证码，这将极其危险——他们仅凭用户名就能登录任意用户的账户，甚至无需知晓用户的密码。



#### 暴力破解双重验证码

由于验证码经常为四位或六位，所以比密码还更好暴力破解。

某些网站试图通过在用户输入错误验证码达到一定次数时自动注销其账户来防范此类攻击。但这种方法在实践中效果甚微，因为高级攻击者甚至能通过为burp intruder创建宏(macros)来自动化这个多步骤过程。

> 除了 **Turbo Intruder** 之外，绕过 OTP 错误次数限制的常见方法还有：
>
> 1. **Burp Macro + Intruder 自动登录**
> 2. **Python 脚本自动化**
> 3. **多 session 轮换**
> 4. **多 IP 爆破**
> 5. **高并发攻击**
> 6. **寻找 OTP 逻辑漏洞（最常见）**





### 其他身份验证机制中的漏洞

例如，用户通常可以修改密码或在忘记密码时重置密码。这些机制也可能引入漏洞，被攻击者利用。

网站通常会谨慎规避登录页面的已知漏洞，但是经常忽略，攻击者能够创建自己的账户，从而轻松访问这些附加页面。

 

#### 保持用户登录

一个常见功能是**允许用户在关闭浏览器会话后仍保持登录状态**。这通常是一个简单的复写框，标注为“`remember me`”或“`keep me logged in`。

此功能通常生成某种 `remember me` 的令牌实现，该令牌会存储在持久性Cookie中。由于持有此Cookie可直接绕过整个登录流程，因此最佳实践是使该Cookie难以被猜出。然而**某些网站会基于可预测的静态值组合(如用户名和时间戳)生成此cookie。也可能将密码直接纳入cookie。**若攻击者能够创建自己的账户，便可分析自身cookie推导生成规则。

或者：某些网站对cookie进行某种加密处理，即使使用静态值也无法被猜出。虽然争取实施时可能有效，但若仅使用base64这类简单的双向编码来加密”cookie“，则完全无法提供任何保护。

即便采用单向哈希函数进行正规加密也并非绝对安全。**若攻击者能轻易识别哈希算法且未添加盐值，它们只需要对字典表进行哈希处理即可暴力破解cookie，当登录次数限制未同步应用于cookie猜测时，此方法被用于绕过登录限制机制。**

若攻击者不能创建自己的账户，仍可能利用此漏洞：通过常规技术(如XSS)，攻击者可**窃取其他用户的cookie**，并据此推断cookie的构造方式。若网站采用开源框架构建，cookie构造的关键细节甚至可能被公开记录。

在少数情况下，即使密码经过哈希处理，仍有可能从cookie中获取明文密码。网上有知名密码列表的哈希版本



#### 重置用户密码

在实际应用中，很多人都会忘记密码，所以提供密码重置功能也是很常见的。

##### 提供电子邮件发送密码

有些网站会生成新密码，并通过电子邮件发送给用户，一般而言，应避免通过不安全的渠道传输持久性密码。在此情况下，安全性依赖于生成的密码在极短的时间内失效，或用户立即更改密码。否则，此方法极易遭受中间人攻击。                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       

##### 通过URL重置密码

更可靠的密码重置方法是向用户发送一个独特的URL链接，引导其进入密码重置页面。安全性较低的实现方式会使用包含易于猜测的参数的URL来识别待重置的账户：

```
http://xxx.com/reset-password?user=victim-user
```

如果把user换成已知的任意用户名，随后他们将直接跳转到可为改任意用户设置新密码的页面。

更优的实现方案就是生成高熵的令牌。当用户访问此URL时，系统应在后端验证该令牌是否存在，若存在则无需确认对应的用户密码重置请求。并且该令牌应在短时间失效。

然而，某些网站在重置表单提交时，未能再次验证令牌。

**若重置邮件中的URL是动态生成的，**则可能存在密码重置中毒漏洞。此时攻击者可能窃取其他用户的令牌，并利用其更改密码。



##### 更改用户密码

密码修改功能若允许攻击者在未登录受害者账户的情况下直接访问，则存在严重安全隐患。例如，当用户名以隐藏字段形式提供时，攻击者可能通过修改请求中的该值来针对任意用户发起攻击。此类漏洞可能被利用来枚举用户名并实施爆破攻击。



#### 密码重置中毒

##### 密码重置是如何进行的？

1. 用户输入用户名或电子邮件地址，并提交密码重置请求。
2. 网站会验证该用户是否存在，然后生成一个临时，唯一且熵值高的令牌，并将该令牌与后端用户账户进行关联。
3. 该网站会向用户发送一封包含密码重置链接的电子邮件。用户的唯一重置令牌作为查询参数包含在相应的URL中：`http://xx.com/reset?token=...`
4. 当用户访问此网址时，网站会验证提供的令牌是否有效，并据此确定需要重置哪个账户。如果一切正常，系统将提示用户输入新密码，最后，令牌销毁。

其安全性依赖这一个原则：**只有目标用户才能访问自己的电子邮件收件箱，从而获得唯一的令牌。密码重置中毒是一种通过窃取该令牌来更改其他用户密码的方法。**

##### 如何构建密码重置中毒攻击？

如果发送给用户的URL是根据可控输入（例如HOST标头）动态生成的，则可能存在如下构建密码重置中毒攻击的风险：

1. 攻击者获取受害者的电子邮件地址或用户名（不确定），并冒充受害者提交密码重置请求。在提交表单时，他们拦截生成的HTTP请求，并修改Host头部，使其指向他们控制的域名。
2. 受害者会直接收到来自该网站的真实密码重置邮件。这封邮件看似包含一个用户重置密码的普通链接，其中还包含一个与该账户关联的有效密码重置令牌。然而，URL中的域名却指向的攻击者的服务器：`https://evil-user.net/reset?token=0a1b2c3d4e5f6g7h8i9j`
3. 如果受害者点击此链接（或该链接通过其他方式被调用，例如杀毒软件扫描器调用）密码重置令牌将发送到攻击者的服务器。
4. 攻击者现在可以访问存在漏洞的网站的真实网址，并通过相应的参数提交从受害者那里窃取的令牌。随后，他们便能将用户的密码重置为任意内容，并登录该账户。

**即使无法控制密码重置链接，有时候也可以利用Host标头向敏感邮件中注入HTML。请注意，电子邮件客户端通常不会执行JavaScript，但其他HTML注入技术可能有效。**
